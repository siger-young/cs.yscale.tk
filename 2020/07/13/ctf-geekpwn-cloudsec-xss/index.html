[GeekPwn 2020] 云安全热身赛两道 XSS 题解

Posted on 2020-07-13 In Security , Web Security

GeekPwn 2020 的云安全热身赛结束了，一共 2 天，把能做的 Web 都做了，比赛的 Web 题考察面比较古怪，让我这种 Web 狗感到极度不舒适. 一共 4 道 Web 题，一题签到类的 Web 题，一题假 Web 题（rtmpsrv 0day emmm….），剩下两道 XSS 相关的题我做了出来，其中一道颇为巧妙，并且有潜在的用途，写个文章记录一下.

noxss 2020

题目概述

本题目有两个服务：

  • 服务 1：http://noxss2020.cal1.cn:3000/
  • 服务 2：http://noxss2020-bot.cal1.cn:1337/

服务 1 运行着一个 flask 应用，源代码 app.py 已经由题目附件给出

限制了 IP 访问，可以看到 flag 位于一个字典中，相关的静态资源在本题中没有太大用处，故不再提及.

服务 2 是一个 BOT，其 IP 对应了上方的 getenv('BOT_IP')：

它会通过 selenium 无头浏览器访问用户输入的 URL，而我们需要做的就是传入一个链接，利用 BOT 访问它，来拿到服务端应用的 flag.

题目思路

首先在上述 app.py 中，我们容易发现一个事实：跨域策略十分苛刻，因此将其通过 iframe 引入是不可能的；接着注意到路由 /search 的功能，是针对参数 keyword 在字典中查找所有包含 keyword 的值，过滤成一个新字典输出，如果没有找到，状态码返回 404，

访问一下字典里的链接，发现可能是命题人留下的提示.

链接 2017

对于字典中 key 为 2017 的一项，我们发现它是一道利用 CSS 注入进行内容嗅探的 XSS 题的 writeup，但是对于本题而言，没有任何注入点，因此绝对不是这个方法.

链接 2019

对于字典中 key 为 2019 的一项，也是一个 XSS 题的 writeup，利用 CSS 的 [attr^=value] 选择器进行前缀查找并设置宽度样式，使得页面出现滚动条，随即再设置滚动条的样式，其中利用了 background 和 url 来达到远程请求，但是对于本题而言，没有任何办法将目标的 /search 页面嵌入网页内，因此也绝对不是这个方法.

两个链接的用意

那么为什么要放这两个链接呢？事实上，我在做完之后仔细琢磨，发现这两道题分别给出了一个线索：

  • link 标签
  • 前缀查找

绕过其中一个限制

我们注意到，app.py 发送的一个限制头是 X-Content-Type-Options: nosniff，事实上这一限制可直接绕过：在 Chrome 系的浏览器 （这里 BOT 的 selenium 驱动应该是 Chromium 的），出于某些原因，大部分资源不会直接受到目标 X-Content-Type-Options: nosniff 的约束，但是由于默认的安全策略，仍然会被 block，但不会产生 error，

潜在用途

基于此利用客户端的计算资源，对某些网站进行 URL 扫描，如果未来这个跨域问题没有解决的话，应该还是有很大利用空间的.

umsg

题目概述

本题目同样有两个服务：

  • 服务 1：http://umsg.iffi.top:3000/
  • 服务 2：http://umsg-bot.iffi.top:2333/

服务 1 运行着一个 Vue 前端，似乎没有和后端通信的操作.

服务 2 是一个 BOT，仍然使用 selenium.

其会携带一个 cookie 访问输入的 URL，而这个 cookie 内容就是 flag，当然，该 cookie 只作用在 umsg.iffi.top 上，因此需要 XSS 实现信息盗取.

题目思路

这个前端肯定藏了东西，直接把整个页面相关的静态文件都 dump 下来，因为这些脚本都是 webpack 的 bundle，直接利用 Prettier 美化一下，然后审计，一开始觉得这边没有后端交互的话，可能是前端藏着什么路由，且该路由有 XSS 注入点，结果怎么找也找不到.

再三地审计，发现了一点蛛丝马迹，位于 index 的 bundle 中

这边有监听 message 事件，并且其具备接受 postMessage 发送的消息的能力，此时就豁然开朗了，我们只需要把这个网页放到一个框架中即可与之通信，不过并不能高兴的太早，这里对 origin 做出了限制，乍看好像只能限制在 umsg.iffi.top 这个域下，实际上只要域名以 umsg.iffi.top 开头就可以了，即刻建一个子域名.

这里还是要理清楚一下思路的，因为 cookie 只在框架中生效，因此需要我们能让框架执行任意代码，或者说让框架把框架内的 document.cookie 利用 postMessage 回传回来.

因此除了主页面需要向框架 postMessage 以载入 payload 之外，还需要框架 postMessage 将相关结果带回.
